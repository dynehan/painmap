# 빵지도 — 대전 베이커리 동선 추천 서비스

## 1. 한 줄 소개

빵투어를 계획할 때, 여러 빵집을 고르면 가장 효율적인 방문 순서를 자동으로 계산해주는 웹 서비스.

## 2. 문제 정의

대전에서 빵투어를 하다 보면 다음과 같은 문제를 겪는다.

- 동선을 고려하지 않고 방문 순서를 짜서 이동 시간을 낭비한다.
- 마감시간을 놓치거나, 마감시간에 맞춰 방문해도 원하는 빵이 이미 품절된 경우가 많다.

기존 지도 앱(카카오맵, 네이버지도 등)은 개별 장소 검색과 길찾기는 지원하지만, **여러 장소를 한 번에 고려한 최적 방문 순서**는 제공하지 않는다.

## 3. 타겟 사용자

- 대전에서 여러 빵집을 하루에 돌아보고 싶은 빵 애호가
- 한정된 시간 안에 효율적으로 여러 곳을 방문하고 싶은 사람

## 4. 핵심 기능

### 4.1 동선 자동 설계 (핵심, AI 에이전트 요소)

- 사용자가 지도 또는 리스트에서 여러 빵집을 선택
- 선택한 빵집들을 방문하기 가장 효율적인 순서 **상위 3개(서로 다른 경로)**를 자동 계산
- 결과를 "A → C → B 순으로 방문하는 것이 효율적입니다" 형태로 제시, 1순위를 선택하면 지도 위 경로 색상과 리스트 항목 색상을 일치시켜 표시
- 시작 지점은 고정하지 않고, 선택한 빵집들 사이에서 순서만 최적화
- **이동수단별 소요시간**(도보/자동차/버스) 비교 제공
  - 1차: 거리 기반 근사치로 시뮬레이션 (도보/차량/버스 속도만 다르게 계산)
  - 2차(시간 여유 시): 실제 지도·대중교통 API로 교체하여 정확도 향상

### 4.2 로그인 / 회원가입 (핵심)

- 아이디·비밀번호 기반 로그인/회원가입
- 회원가입 시 빵 취향, 가본 곳, 가고 싶은 곳 선택 항목 포함 (추후 큐레이터 기능과 연동 가능)
- 마이페이지에서 내 정보 및 취향 확인/수정

### 4.3 대화형 빵집 큐레이터 (부가 기능, 시간 여유 시 추가)

- 자연어 요청("달달한 거 먹고 싶어", "조용한 분위기 원해")에 맞춰 LLM이 보유 데이터 중 적합한 곳을 추천
- 네비게이션에 "챗봇" 메뉴로 자리는 마련해두되, 기능 자체는 여전히 부가 기능으로 취급 — 4주차까지 다른 핵심 기능이 안정화된 뒤 시간이 남을 때 구현

## 5. 데이터 설계

**수집 범위**: 대전 전역, 목표 15~20곳 (4주차 데모 기준)

**빵집 1곳당 수집 항목**
| 항목 | 설명 |
|---|---|
| 이름 | 상호명 |
| 위치 | 위도/경도, 주소 |
| 대표메뉴 | 유명한 빵 종류 |
| 한줄 코멘트 | 직접 방문/조사 기반 코멘트 |
| 가격대 | 대략적인 가격 범위 |
| 영업시간 | 오픈/마감 시간 |
| 휴무일 | 정기 휴무 요일 |
| 사진 | 대표 이미지 1장 |
| 몰리는 시간 | 붐비는 시간대 (직접 조사 또는 방문 후기 기반 추정) |

**데이터 소스**

- 공공데이터포털 제과점 영업 데이터 (기본 정보: 상호명, 주소)
- 대표메뉴/코멘트/가격대/몰리는 시간 등 큐레이션 정보는 직접 조사하여 입력

**사용자 데이터**
| 항목 | 설명 |
|---|---|
| 아이디/비밀번호 | 로그인용 계정 정보 |
| 빵 취향 | 선호하는 빵 종류(체크리스트) |
| 가본 곳 | 방문한 빵집 목록 |
| 가고 싶은 곳 | 찜한 빵집 목록 |

## 6. 사용자 시나리오

다인이는 주말에 대전 대학가에서 빵투어를 하기로 했다.

앱을 켜니 첫 화면(홈)에 지도가 뜨고, 근처 베이커리들이 마커로 표시된다. 마커를 몇 개 눌러보다가, 이름과 대표메뉴를 한눈에 비교하고 싶어져 하단 탭에서 리스트 뷰로 전환한다. 리스트에서 성심당 본점, 하레하레, 앤드유 세 곳을 체크한다 — 지도로 다시 돌아가 보니 방금 고른 세 곳의 마커가 선택된 상태로 표시되어 있다.

화면 하단에 뜬 "선택함 3" 배지를 눌러 선택함으로 이동한다. 목록을 한 번 더 확인하고, 마음이 바뀐 한 곳을 목록에서 빼려다 그대로 두기로 하고 "동선 계산하기"를 누른다.

결과 화면에 "성심당 본점 → 앤드유 → 하레하레" 순으로 방문하라는 추천과 함께 "이 순서로 방문하면 이동 거리를 최소화할 수 있어요"라는 한 줄 설명이 뜬다. 다인이는 별다른 고민 없이 그 순서 그대로 빵투어를 시작한다.

## 7. 화면 구성

### 7.1 화면 목록

| 화면         | 설명                                      |
| ---------- | --------------------------------------- |
| 홈 (지도)     | 서비스 접속 시 첫 화면. 대전 지역 베이커리를 지도 위에 마커로 표시 |
| 리스트 뷰      | 지도와 연동된 목록형 화면                          |
| 빵집 상세      | 마커 클릭 시 표시. 위치, 연락처, 대표 빵, 몰리는 시간 등     |
| 경로 결과 화면   | 2곳 이상 선택 시 진입. 상위 3개 경로와 이동수단별 소요시간 제공  |
| 챗봇         | 대화형 큐레이터 (부가 기능)                        |
| 마이페이지      | 내 정보, 빵 취향, 가본 곳/가고 싶은 곳                |
| 로그인 / 회원가입 | 계정 생성 및 로그인, 회원가입 시 취향 설문 포함            |

웹 앱(데스크톱 브라우저 기준) 형태로, 상단 검색바 + 하단 메뉴바(지도-리스트-챗봇-마이페이지) 구조를 기본으로 한다.

### 7.2 화면별 상세 구조

**홈 (지도)**

- 상단: 검색 바 + 로그인/회원가입 버튼(비로그인 시)
- 중앙: 카카오맵 위에 빵집 마커 표시
- 하단: 지도 - 리스트 - 챗봇 - 마이페이지 메뉴바

**빵집 상세**

- 마커 클릭 시 표시: 위치, 전화번호, 대표 빵, 몰리는 시간
- 하단: "선택하기" 버튼
- 2곳 이상 선택되는 순간 자동으로 경로 결과 화면으로 전환

**경로 결과 화면**

- 상단 30%: 지도 위에 경로를 색으로 표시
- 하단 70%: 순위별 경로 카드 (베이커리 개수에 따라 최대 3개, 서로 다른 순서 조합 중 이동거리가 짧은 순)
  - "가장 빠른 경로" 카드를 클릭하면 상세 리스트로 펼쳐짐
  - 리스트는 방문 순서대로 정렬되고, 각 항목 박스 색상이 지도 위 해당 지점 색상과 일치 (예: (1) 하레하레 — 파랑, (2) 성심당 본점 — 빨강)
- "전체 이동 시간 보기" 버튼
  - 클릭 시 버스 / 자동차 / 도보 탭 등장, 이동수단별 소요시간을 다르게 표시
  - 1차 구현은 거리 기반 근사치, 이후 실제 API로 교체

**로그인 / 회원가입**

- 로그인: 아이디, 비밀번호
- 회원가입: 아이디, 비밀번호, 빵 취향(체크리스트), 가본 곳/가고 싶은 곳 선택

**마이페이지**

- 로그인한 사용자의 정보, 빵 취향, 가본 곳/가고 싶은 곳 목록 확인 및 수정

**챗봇**

- 자연어로 빵집을 추천받는 대화형 화면 (부가 기능)

## 8. 기술 스택

- **프론트엔드**: React (Vite) 기반 웹 앱 (모바일 전용이 아닌 데스크톱 브라우저 중심 레이아웃)
- **지도**: 카카오맵 JavaScript SDK
- **백엔드/DB**: 직접 구축 (Node.js 기반 API 서버 + DB) — 이번 프로젝트를 통해 백엔드 개발 경험을 쌓는 것이 목표 중 하나
- **인증**: 아이디/비밀번호 기반 로그인, 비밀번호는 해시하여 저장. 세션 또는 토큰 기반 인증 중 구현 단계에서 선택
- **동선 계산 로직**:
  - 빵집 개수가 적을 때(대략 8곳 이하): 모든 순서 조합을 계산하는 완전탐색 방식으로 정확한 최적해 도출, 그중 서로 다른 상위 3개 경로 제시
  - 빵집 개수가 많을 때: 가까운 곳부터 순서대로 잇는 휴리스틱 방식으로 근사해 도출 (실행 속도 확보)
- **이동수단별 소요시간**:
  - 1단계: 거리 기반 근사치로 시뮬레이션 (도보/차량/버스 속도 계수만 다르게 적용)
  - 2단계(시간 여유 시): 카카오모빌리티/대중교통 API 등 실제 데이터로 교체

## 9. 리스크 및 대응

| 리스크                                            | 대응                                                                     |
| ---------------------------------------------- | ---------------------------------------------------------------------- |
| 백엔드 직접 구축에 예상보다 시간이 오래 걸릴 수 있음                 | 데이터 목표치를 30곳 이상에서 15~20곳으로 하향 조정하여 시간 확보                               |
| 로그인/인증까지 더해지며 범위가 커짐                           | 아이디/비밀번호 기반의 최소 기능으로 한정 (소셜 로그인, 비밀번호 찾기 등은 이번엔 제외)                    |
| 실제 대중교통 API 연동은 난이도가 매우 높음                     | 1차는 거리 기반 근사치로 구현, 실제 API는 시간이 남는 4주차에 시도. 안 되더라도 근사치 버전으로 데모 가능하도록 설계 |
| 경로 3개(서로 다른 순서)를 보여주는 로직의 복잡도                  | 완전탐색 결과를 거리순으로 정렬한 뒤, 겹치는 순서(정방향/역방향 등)를 제외하고 상위 3개만 추출하는 방식으로 단순화     |
| 선택 개수가 늘어나면 동선 계산이 느려질 수 있음                    | 일정 개수 이상은 완전탐색 대신 휴리스틱 알고리즘으로 전환                                       |
| 데이터 큐레이션(가격대, 영업시간, 몰리는 시간 등)에 시간이 많이 소요될 수 있음 | 1주차 안에 하루 평균 큐레이션 가능 개수를 체크하여, 필요 시 목표치 재조정                            |
| 지도/리스트/챗봇/마이페이지까지 화면이 많아짐                      | 챗봇은 끝까지 부가 기능으로 유지, 다른 화면이 안정된 이후에만 착수                                 |

## 10. 4주 로드맵 (초안)

| 주차  | 목표                                                              |
| --- | --------------------------------------------------------------- |
| 1주  | 백엔드 API/DB 기본 구조 설계 및 구축(회원 테이블 포함), 카카오맵 연동, 데이터 수집 착수 및 속도 체크 |
| 2주  | 로그인/회원가입 구현, 빵집 선택 → 동선 계산(상위 3개 경로) 로직 구현, 경로 결과 화면 완성         |
| 3주  | 이동수단별 소요시간(1차: 근사치) 구현, 마이페이지, UI 다듬기                           |
| 4주  | 버그 수정, 시간 남으면 실제 대중교통 API 교체 및 챗봇 부가기능, 데모/발표 준비                |
