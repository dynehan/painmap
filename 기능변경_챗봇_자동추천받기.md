# 기능 변경: 챗봇 → 자동 추천받기 (v3 — 조건 분리/인증 전제 반영)

v1에서 클로드코드가 지적한 데이터·구현 불일치를 반영해 범위를 다시 정리했습니다(v2).
v2 재검토에서 추가로 발견된 것 반영(v3): 이동수단을 빵집 선정 조건에서 분리, 로그인 검증 미들웨어 선행 필요, 3번 질문 키워드 매칭을 다중 키워드 방식으로 보완.

## 배경

기존 "대화형 빵집 큐레이터(챗봇)"는 자연어로 취향을 물어보는 방식인데, 다음 이유로 선택형 퀴즈 방식으로 대체합니다.

- 회원가입 취향 설문, 리스트 필터와 역할이 겹쳐서 챗봇만의 존재 이유가 약함
- "대전 빵집을 전혀 모르는 사람"에게는 자유 대화보다 명확한 선택형 질문이 더 도움이 됨
- LLM 연동 없이 버튼 선택 + 규칙 기반 필터링만으로 구현 가능해 공수·리스크가 낮음

## 변경 요약

- 챗봇 화면과 하단 네비게이션의 "챗봇" 탭 제거
- 지도/홈 화면에 **"자동 추천받기"** 버튼 배치 (비로그인 사용자도 사용 가능)
- 클릭 시 선택형 질문 → 조건에 맞는 빵집 자동 선정 → 서버에서 동선 계산까지 끝낸 결과를 바로 경로 결과 화면에 표시

## 질문 흐름 (v1 범위)

| 순서  | 질문  | 선택지 | 활성화 조건 |
| --- | --- | --- | --- |
| 1   | 몇 곳 돌고 싶어요? | 2곳 / 3곳 / 4곳 (정확히 4곳으로 고정, "이상" 아님) | 항상  |
| 2   | 이동수단은? | 도보 / 자동차 / 버스 | 항상  |
| 3   | 어떤 빵이 좋아요? | 크루아상 / 식빵 / 단팥빵 / 스콘 / 바게트 (회원가입 취향 태그와 동일 값으로 통일) | **`bakeries.signature_menu`에 이 키워드가 채워진 곳이 충분히 쌓인 뒤에만 노출.** 지금은 대부분 빈 값이라 v1에서는 UI에서 숨김 |
| 4   | 가본 곳은 뺄까요? | 예 / 아니오 | **`user_bakery_status`에 실제 데이터가 쌓이기 시작한 뒤(3주차 목표 4번: 마이페이지 실데이터 연동 완료)** + **서버에 로그인 검증 미들웨어가 생긴 뒤**에만 노출. 그 전엔 숨김 |

→ 즉 지금 당장 구현 가능한 v1은 질문 1, 2만으로 진행하고, 3·4번은 각각의 선행 데이터 작업이 끝나면 켜는 기능 플래그로 둡니다.

## 데이터/로직 흐름

1. 프론트: 활성화된 질문에만 답변 받아 조건 객체 구성.
   - **빵집 선정 조건**: `{count, style?, excludeVisited?}` — 이동수단(2번 질문)은 여기 안 들어간다.
   - **이동수단(2번 질문)**: 빵집을 뭘 고를지와는 무관하고, 방문 순서 사이 이동시간을 어떤 속도(도보/자동차/버스)로 보여줄지의 문제다. 서버 필터링에 넘기지 않고, 경로 결과 화면에 진입할 때 "이동 거리 보기" 패널의 초기 선택 탭으로만 반영한다.
2. 서버 — 신규 엔드포인트 `POST /api/recommend`:
  - `{count, style?, excludeVisited?}` 조건에 맞는 빵집을 좌표(lat/lng) 포함해서 DB에서 조회
  - 조건에 맞는 곳이 `count`보다 적으면 아래 순서로 조건을 완화 (fallback):
    1. 스타일 조건 해제 → 2) 가본 곳 제외 조건 해제 → 3) 그래도 부족하면 있는 만큼만 사용
  - 조회한 빵집 목록을 **routeService의 계산 함수에 서버 내부에서 바로 전달**해 동선 계산까지 끝냄 (`/api/routes`를 프론트가 다시 호출하는 게 아니라, `/api/recommend` 응답 자체가 이미 계산된 상위 3개 경로)
3. 프론트는 응답을 받아 바로 경로 결과 화면으로 이동, 2번 질문 답변으로 이동수단 탭을 미리 선택해둔다.

### 3번 질문 키워드 매칭 방식

`signature_menu`는 자유 텍스트라 카테고리명을 그대로 부분일치 검색하면 놓치는 경우가 있다(예: 정인구팥빵의 메뉴는 "팥빵"이라 "단팥빵"으로 검색하면 안 걸림 — 포함 관계가 반대 방향). 카테고리 하나당 키워드를 여러 개 매핑해서 매칭한다.

```
크루아상: ['크루아상']
식빵: ['식빵']
단팥빵: ['단팥빵', '팥빵']
스콘: ['스콘']
바게트: ['바게트']
```

이 매핑은 데이터가 늘어날 때마다 같이 다듬어야 하는 항목이라, 하드코딩된 상수로 두고 위치를 명확히 남겨둔다(`server/src/services/recommendService.js` 상단 등).

**주의**: 이 자동 전환은 퀴즈 결과에만 적용되는 예외입니다. 오늘 되돌린 "수동 선택 2곳 이상 시 자동 전환" 동작을 다시 살리는 게 아니라, 수동 선택(트레이 클릭 확인) 흐름은 지금 상태 그대로 둡니다. 퀴즈는 애초에 "결과를 통째로 갖다주는" 성격이라 별도 흐름으로 취급합니다.

## `/api/routes`와의 관계

기존 `/api/routes`는 `{id, lat, lng}` 배열을 받아 순수 계산만 하는 구조를 그대로 유지합니다. `/api/recommend`는 이 계산 함수를 내부적으로 재사용하되, DB 조회부터 계산까지 한 번에 끝내는 별도 엔드포인트로 추가합니다 (프론트가 두 번 호출할 필요 없게).

## 영향받는 문서/파일

- 기획서 4.3 "대화형 빵집 큐레이터" → "자동 추천받기"로 교체
- 화면 목록(7.1)에서 "챗봇" 행 제거 (결과는 기존 경로 결과 화면 재사용)
- `CLAUDE.md` 디렉토리 구조의 `ChatScreen` → `AutoRecommendScreen`(가칭)으로 교체
- 추가기능 스펙 문서의 챗봇 관련 항목 삭제
- (신규) `server/src/routes/recommend.js`, 기존 `routeService.js` 재사용
- (신규 선행 작업) `bakeries.signature_menu`에 취향 키워드 태깅 — 3번 질문 활성화 전제조건. 이건 이번주 계획 1번(빵집 실데이터 수집) 범위에 포함해서 진행
- **(신규 선행 작업) JWT 로그인 검증 미들웨어** (`server/src/middleware/auth.js` 등) — 지금 서버엔 "이 요청이 어느 유저 건지" 확인하는 장치가 없음. 지금까지는 보호가 필요한 API가 없어서 안 만들었는데, 4번 질문(가본 곳 제외)이 이 기능의 첫 사용처가 됨. 마이페이지 실데이터 연동과 함께 진행하는 게 맞음

## 결정된 세부사항

- 1번 질문은 "이상"이 아니라 2/3/4곳 중 정확히 하나를 고르는 방식
- 이동수단(2번 질문)은 빵집 선정 조건이 아니라 결과 화면의 초기 이동수단 탭 선택에만 쓰임
- 조건에 맞는 곳이 부족할 때 fallback 순서: 스타일 해제 → 가본곳 제외 해제 → 있는 만큼 사용
- API 연결 방식: 추천 엔드포인트가 DB 조회 + 동선 계산을 한 번에 끝내서 결과만 반환 (프론트의 추가 호출 없음)
- 3번 질문 키워드 매칭은 카테고리당 다중 키워드 매핑으로 처리 (단일 정확 매칭 아님)
- 4번 질문은 로그인 검증 미들웨어가 생긴 뒤에만 활성화

## 우선순위

부가 기능(4주차 여유 시) 범주는 유지하되, 질문 1·2만으로 이루어진 v1은 3주차 안에 다른 백로그(빵집 데이터, 리스트 API, 동선 서버 이전)가 끝난 뒤 시도해볼 만한 수준으로 공수가 낮습니다. 질문 3·4는 각 선행 작업이 끝나기 전까지 백로그에 남겨둡니다.
